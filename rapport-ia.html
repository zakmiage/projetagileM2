Rapport d'Utilisation de l'Intelligence Artificielle dans le Projet

Projet KUBIK ERP — M2 Ingénierie Logicielle · UE Projet Agile · Avril 2026
Équipe : Dardan, Zacharie, Arno, Huzaifah, Oumeyma, Zakaria, Youba
L'assistant IA Gemini (Google DeepMind) a été intégré comme outil d'assistance au développement dans le respect d'un cadre éthique strict : supervision humaine systématique, périmètre délimité aux tâches à faible valeur ajoutée, et usage exclusivement via interface conversationnelle — jamais en mode agent autonome.

1. Cadre d'utilisation

L'équipe a défini dès le démarrage un protocole garantissant que l'IA reste un outil de conseil et non un acteur décisionnaire. Toute suggestion est soumise à validation humaine explicite avant intégration, et aucune commande (commit, migration SQL, installation de dépendance) n'est exécutée sans approbation manuelle. L'interface conversationnelle retenue exclut par construction tout mode d'exécution autonome : le code proposé est copié, analysé et intégré manuellement par le développeur.

2. Usages concrets dans le projet

  • Génération de données de test : seeds SQL (shifts, kanban, budget) et 11 factures PDF réalistes via Python/xhtml2pdf — tâches répétitives sans valeur ajoutée intellectuelle directe.
  • Débogage assisté : analyse du bug R14 (mélange forecast/actual dans syncFsdieSubvention), du bug capacity||10 (opérateur falsy traitant 0 comme 10), et de l'erreur NG0203 (conflit Service Worker). Chaque correctif proposé a été relu, compris et testé avant application.
  • Documentation technique : rédaction initiale de la SFD et STD par rétro-ingénierie du code source, mise à jour de la v1.1 à la v1.7. Chaque version relue et corrigée par l'équipe avant commit.
  • Optimisation : proposition de l'architecture de short polling (5 s), cache mémoire Map<eventId> et mise à jour optimiste dans le module Budget. Adapté à l'architecture Angular existante avant intégration.

3. Garde-fous et supervision

Ce que l'IA peut faireCe que l'IA ne fait pas
Suggérer du code soumis à validationPrendre des décisions architecturales
Générer des jeux de données de testExécuter une action sans approbation explicite
Analyser des erreurs et proposer des correctifsFusionner ou pousser du code sur les branches
Rédiger la documentation à partir du codeRemplacer la revue de code entre pairs
Vérifier la cohérence des règles métierAccéder à des données réelles ou sensibles

Processus systématique suivi : (1) identification de la tâche → (2) requête contextuelle → (3) proposition IA → (4) revue critique → (5) approbation explicite → (6) test avant merge.

4. Limites reconnues

  • Hallucinations : plusieurs suggestions incorrectes (colonnes SQL inexistantes, chemins de fichiers erronés) ont été détectées lors des tests, confirmant que l'étape de validation ne peut être sautée.
  • Dépendance potentielle : risque de sollicitation réflexe, pallié en limitant les usages aux tâches clairement identifiées en amont et en exigeant que le développeur soit capable d'expliquer chaque modification intégrée.
  • Périmètre incompressible : l'IA est inefficace sur les dimensions à forte valeur ajoutée — arbitrages métier, dynamique d'équipe, choix d'architecture long terme — qui sont restées entièrement sous responsabilité humaine.

5. Conclusion

L'IA a permis d'accélérer les tâches répétitives et de libérer du temps pour les problèmes à forte valeur ajoutée. La discipline imposée — validation systématique, interface conversationnelle uniquement, périmètre défini — démontre qu'une utilisation responsable est possible en contexte académique. L'humain reste décisionnaire, l'IA conseille et assiste. L'ensemble de l'équipe confirme que ces principes ont été respectés tout au long du sprint, et qu'aucun code n'a été intégré sans compréhension et validation humaine préalable.
