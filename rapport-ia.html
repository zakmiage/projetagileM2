Rapport d'Utilisation de l'Intelligence Artificielle dans le Projet

Projet KUBIK ERP — Gestion d'événements associatifs
M2 Ingénierie Logicielle · UE Projet Agile · Équipe : Dardan, Zacharie, Arno, Huzaifah, Oumeyma, Zakaria, Youba · Avril 2026
L'assistant IA Gemini (Google DeepMind) a été intégré comme outil de soutien au développement dans le respect d'un cadre éthique strict : supervision humaine systématique, périmètre délimité aux tâches à faible valeur ajoutée, et utilisation exclusivement via interface conversationnelle — jamais en mode agent autonome.

1. Cadre d'utilisation

Dès le démarrage, l'équipe a défini un protocole garantissant que l'IA reste un outil d'assistance et non un acteur décisionnaire. Toute suggestion est soumise à une validation humaine explicite avant intégration, et aucune commande (commit, migration SQL, installation de dépendance) n'est exécutée sans approbation manuelle.

L'outil retenu — Gemini via interface conversationnelle — exclut par construction tout mode d'exécution autonome : le code proposé doit être copié, analysé et intégré manuellement par le développeur.

2. Usages dans le projet

  • Jeux de données de test : génération des seeds SQL (shifts, kanban, budget) et de 11 factures PDF réalistes via Python/xhtml2pdf — tâches répétitives sans valeur ajoutée intellectuelle.
  • Débogage assisté : analyse du bug R14 (mélange forecast/actual dans syncFsdieSubvention), du bug capacity||10 (opérateur falsy), et de l'erreur NG0203 (Service Worker). Chaque correctif proposé a été relu et testé avant application.
  • Documentation : rédaction initiale de la SFD et STD par rétro-ingénierie du code, mises à jour de la v1.1 à la v1.7. Chaque version relue et corrigée par l'équipe.
  • Optimisation : proposition de l'architecture de short polling (5s), cache mémoire Map<eventId> et mise à jour optimiste. Adapté à l'architecture Angular existante avant intégration.

Le processus systématique suivi : (1) identification de la tâche → (2) requête contextuelle → (3) proposition IA → (4) revue critique du développeur → (5) approbation explicite → (6) test avant merge.

4. Limites reconnues

  • Hallucinations : plusieurs suggestions incorrectes (colonnes SQL inexistantes, chemins erronés) ont été détectées lors des tests — confirmant que l'étape de validation ne peut être sautée.
  • Dépendance : risque de sollicitation réflexe, pallié en limitant les usages aux tâches clairement identifiées en amont.
  • Compréhension du contexte : l'IA est inefficace sur les dimensions à forte valeur ajoutée (arbitrages métier, dynamique d'équipe, choix long terme) — dimensions restées entièrement humaines.

5. Conclusion

L'IA a accéléré les tâches répétitives et libéré du temps pour les problèmes complexes. La discipline imposée — validation systématique, interface conversationnelle uniquement, périmètre défini — démontre qu'une utilisation responsable est possible en contexte académique. L'humain reste décisionnaire, l'IA conseille et assiste.

L'ensemble de l'équipe confirme que ces règles ont été respectées tout au long du sprint. Aucun code n'a été intégré sans compréhension et validation humaine préalable.
